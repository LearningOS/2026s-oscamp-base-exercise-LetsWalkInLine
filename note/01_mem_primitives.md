# 学习笔记：no_std 内存操作原语实现 (Memory Primitives)

在进行操作系统底层开发时，我们处于 `#![no_std]`（无标准库）环境中。这意味着我们无法使用 `std` 提供的丰富功能（如 `String`, `Vec`），只能依赖 `core` 库，甚至需要手动实现 C 语言标准库中最基础的内存和字符串操作函数。

以下是本次练习的核心知识点和踩坑记录总结：

## 1. 裸指针 (Raw Pointers) 与 `unsafe` 操作
在底层实现内存拷贝时，必须操作内存的绝对地址，这需要用到裸指针：
- `*const u8`：不可变字节指针（只读源数据）。
- `*mut u8`：可变字节指针（写入目标内存）。
- **指针偏移**：在 Rust 中，裸指针不能直接像 C 语言那样使用 `src + i` 或 `src[i]`，而是需要使用 `.add(i)` 方法向后移动 `i` 个字节。
- **解引用**：读取（`*src`）或写入（`*dst = 1`）裸指针指向的内存，必须包含在 `unsafe { ... }` 块中。

## 2. 核心陷阱：空指针 (`is_null()`) vs 空字符 (`\0`)
在实现 `my_strlen` 和 `my_strcmp` 时，遇到了引发 `SIGSEGV`（段错误）的经典 Bug。

- **错误认知**：使用 `!s.add(len).is_null()` 来判断字符串是否结束。
- **真相**：
  - `ptr.is_null()` 判断的是**指针的内存地址（门牌号）**是否为 `0x0`。指针不断向后偏移，地址永远不可能为 `0`，导致死循环和内存越界。
  - `*ptr == 0` 判断的是**内存地址里的值（房间里的内容）**是否为 `0`。C 风格的字符串以数值为 `0` 的字节（即 `\0`）作为结束标志。
- **正确做法**：使用 `while *s.add(len) != 0` 来判断是否读取到了字符串的末尾。

## 3. 字符串比较 (`strcmp`) 的不等长问题
不需要提前判断哪个字符串更长。C 风格字符串巧妙利用了 `\0`（数值为 0）。
例如比较 `"abc\0"` 和 `"abcd\0"`：
- 前三个字符相等。
- 比较第四个字符时，前者是 `\0` (数值 0)，后者是 `'d'` (ASCII 码 100)。
- 因为 `0 != 100`，直接返回差值即可得出大小关系。长度不等的情况会被 `\0` 自然处理掉。

## 4. `memmove` 的内存重叠处理与指针比较
`memmove` 必须处理源地址和目标地址可能重叠的情况：
- 如果 `dst > src` 且发生重叠，**必须从后往前拷贝**，否则前面的数据在被读取前就会被覆盖。
- 如果 `dst <= src` 或无重叠，**从前往后拷贝**即可。
- **指针比较的优雅解法**：在 Rust 严格的内存模型下，直接使用 `<` 或 `>` 比较两个裸指针可能触发未定义行为（UB）。更严谨的做法是先将指针转换为 `usize`（纯粹的整数内存地址），再进行数值比较：
  ```rust
  let src_addr = src as usize;
  let dst_addr = dst as usize;
  if src_addr < dst_addr && src_addr + n > dst_addr {
      // 从后往前拷: (0..n).rev()
  }
  ```

## 5. 指针变量的可变性与数据可变性
在实现 `my_strcmp` 时，需要遍历两个只读指针（`*const u8`）：
- **错误示范**：`let mut p1 = s1 as *mut u8;` (强行将只读数据强转为可写，有风险)。
- **优雅解法**：`let mut p1 = s1;`。这里 `mut p1` 表示**指针变量本身是可变的**（即指针可以指向下一个地址：`p1 = p1.add(1)`），但**它指向的数据依然是不可变的**（`*const u8`）。将变量自身的 Mutability 与底层数据的 Mutability 区分开，是编写 Rusty 代码的关键。
