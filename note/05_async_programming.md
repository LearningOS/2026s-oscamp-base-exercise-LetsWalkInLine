# Rust 异步编程学习笔记

本笔记总结了关于 Rust 异步编程核心机制及 Tokio 运行时使用的关键知识点，涵盖了从底层 `Future` 的手动实现到高层并发控制模式。

## 1. 异步底层机制：手动实现 `Future` (01_basic_future)

Rust 的异步代码最终会被编译器转化为实现了 `Future` 特质的状态机。理解其内部机制是编写高效异步代码的基础。

*   **`Future` 特质与 `poll` 方法**：
    `poll` 方法是异步任务推进的核心。执行器（Executor）通过不断调用 `poll` 来驱动任务前进。
*   **状态返回 (`Poll` 枚举)**：
    *   `Poll::Ready(value)`：表示任务已完成，返回最终结果。
    *   `Poll::Pending`：表示任务正在等待某些条件（如 I/O、定时器），当前未能完成，将执行权让出给运行时。
*   **唤醒机制 (`Waker`)**：
    当 `poll` 返回 `Pending` 时，必须确保在未来某个时刻调用 `Context` 中的 `waker().wake_by_ref()`。这会通知运行时重新将该任务放入执行队列中进行下一次 `poll`。
*   **`Pin<&mut Self>`**：
    异步状态机可能包含自引用结构，`Pin` 用于保证对象在内存中的地址不会发生移动。对于简单结构，可以通过 `self.get_mut()` 安全地获取可变引用。

## 2. 并发执行：Tokio 任务调度 (02_tokio_tasks)

仅仅使用 `async/await` 并不能自动实现并发，多个任务的并发执行依赖于运行时（如 Tokio）的任务调度。

*   **`tokio::spawn`**：
    用于将一个异步任务放入 Tokio 的线程池中后台执行，它是实现并发的关键。任务一旦被 `spawn`，就会立刻开始执行，无需等待 `.await`。
*   **`JoinHandle`**：
    `tokio::spawn` 返回的句柄，类似于一张“提货单”。通过 `.await` 这个句柄，可以等待后台任务执行完毕并获取其返回值。
*   **并发模式的最佳实践**：
    实现并发的常见模式是“先批量 Spawn，后批量 Await”。如果在一个循环中边 Spawn 边 Await，会导致任务退化为串行执行。
*   **`async move`**：
    在多线程运行时中，被 `spawn` 的任务可能跨线程执行，因此必须使用 `async move` 转移外部变量（如循环变量）的所有权。

## 3. 任务间通信：异步通道 (03_async_channel)

遵循“不要通过共享内存来通信，而应通过通信来共享内存”的原则，异步通道是不同 Task 之间传递数据的最佳选择。

*   **`mpsc` (Multi-Producer, Single-Consumer)**：
    多生产者单消费者模型。发送端 (`Sender`) 可以被克隆给多个任务，但接收端 (`Receiver`) 只能保留一个。
*   **阻塞与挂起**：
    当有界通道填满时，`send(...).await` 会挂起当前任务；当通道为空时，`recv().await` 会挂起等待新数据。
*   **通道关闭机制（关键点）**：
    **通道只有在所有的发送端 (`tx`) 都被 Drop 后才会关闭**。在“扇入 (Fan-in)”等多生产者模式中，必须记得手动 `drop` 掉主任务中残留的原始 `tx`，否则接收端的 `recv()` 将陷入无限死锁等待。

## 4. 竞争与控制：Select 宏与超时 (04_select_timeout)

在复杂的网络请求和并发场景中，经常需要同时等待多个操作并只取最快的结果。

*   **`tokio::select!` 宏**：
    允许同时轮询多个 Future（分支）。当其中**任意一个**分支率先完成时，立即执行该分支对应的代码。
*   **取消机制 (Cancellation)**：
    `select!` 最大的特点是，一旦某个分支胜出，**其他所有未完成的分支都会被立即 Drop（取消）**。这避免了资源的浪费。
*   **超时控制实现**：
    利用 `select!` 可以轻松实现超时逻辑：让“目标任务”与“定时器任务（如 `tokio::time::sleep`）”进行赛跑。如果定时器先完成，则说明目标任务超时。
