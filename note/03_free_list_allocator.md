# 03. Free-List Allocator (空闲链表分配器) 笔记

## 核心概念复习
在实现 Free-List 分配器时，掌握了以下几个底层系统编程的关键概念：
1. **侵入式链表 (Intrusive Linked List)**：直接利用被释放的空闲内存块本身来存储链表节点的信息（如 `size` 和 `next` 指针），从而避免了在释放内存时还需要额外分配内存来记录节点。
2. **裸指针操作**：练习了大量的 `*mut u8` 和 `*mut FreeBlock` 的强转，以及通过 `ptr.read()` 和 `ptr.write()` 直接读写内存数据。
3. **首次适应算法 (First-Fit)**：在分配时遍历空闲链表，找到第一个满足大小和对齐要求的块即可返回。
4. **LIFO 释放策略 (头插法)**：释放内存时，将其包装为 `FreeBlock` 并插入到空闲链表的头部。

## Bug 记录与反思：视觉错觉引发的 OOM 测试失败

在实现分配器的后备分配逻辑（从 bump 区域分配）时，遇到了一个难以排查的 Bug，导致 `test_oom` 测试无法通过。

**错误代码：**
```rust
let aligned = (cur_next + align - 1) & !(align - 1);
// 这里的 align 只是对齐要求（比如 8），少打了两个字母！
let new_next = align + size; 
if new_next >= self.heap_end {
    return null_mut();
}
```

**问题分析：**
在计算分配后的新 bump 指针位置时，本应该使用对齐后的起始**地址** `aligned` 加上分配大小 `size`，但我错误地写成了对齐**要求** `align` 加上大小。
这导致 `new_next` 计算出来只是一个非常小的数值（例如 8 + 32 = 40）。而在比较 `new_next >= self.heap_end` 时，由于 `self.heap_end` 是一个真实的、非常大的堆顶内存地址，这个条件几乎永远不会成立。因此，即使申请的大小超过了堆的容量，分配器依然错误地认为还有空间，没有返回 `null_mut()`，最终导致 OOM 测试失败。

**正确的代码应为：**
```rust
let new_next = aligned + size; // 使用对齐后的地址
```

### 经验教训：变量命名的“视觉距离”
这个 Bug 的根本原因在于变量名 `align` 和 `aligned` 长得太像了，导致了在编写时产生“视错觉”，少敲了两个字母却很难被肉眼察觉。

**改进方案：**
在编程中，变量命名不仅要准确表达其含义，还要讲究防呆，刻意拉开变量名之间的**“视觉距离”**。
比如在这个场景中：
- 将 `aligned` 改名为 `aligned_addr` 或 `start_ptr`。
- 将 `align` 保持原样或改名为 `align_req`。

这样不仅语义更加明确（区分了“要求”和“实际地址”），在敲击键盘或进行代码审查时，也能有效避免这种低级却极其隐蔽的错误。
